# Общий бэклог MVP

## Backend

### Задача: привести модель данных к документации
Зачем нужна: сейчас backend — прототип, а в `docs/db/schema.md` описана целевая структура. Без этого дальше все endpoint-ы будут расходиться с фронтом и документацией.
Что обязательно прочитать:
- `docs/db/schema.md`
- `docs/architecture/overview.md`
Где в коде работать:
- `backend/app/models/*`
- `backend/app/core/database.py`
Что должно получиться на выходе:
- модели `users/projects/notes/tasks/files/tags` и связующие таблицы соответствуют docs по полям и связям.
Зависит от: нет.
Можно делать параллельно: с подготовкой Electron-архитектуры, с подготовкой frontend API-клиента, с проектированием AI-контракта.
Крайний срок: 5 мая 2026.

### Задача: сделать авторизацию по контракту
Зачем нужна: весь продукт строится на изоляции пользовательских данных.
Что обязательно прочитать:
- `docs/api/auth.md`
- `docs/api/errors.md`
Где в коде работать:
- `backend/app/api/v1/auth.py`
- `backend/app/services/auth.py`
- `backend/app/core/security.py`
Что должно получиться на выходе:
- работают `register/login/me/me patch`, JWT проверяется, ошибки соответствуют контракту.
Зависит от: от базовой модели `users` и JWT/security-слоя.
Можно делать параллельно: с задачей Андрея по форматам ошибок, с задачами Маши и Даши по UI auth-форм, с Electron dev-оберткой.
Крайний срок: 6 мая 2026.

### Задача: реализовать проекты, заметки, задачи, файлы, теги и поиск
Зачем нужна: это ядро MVP по продуктовой модели из docs.
Что обязательно прочитать:
- `docs/api/projects.md`
- `docs/api/notes.md`
- `docs/api/tasks.md`
- `docs/api/files.md`
- `docs/api/tags.md`
- `docs/api/search.md`
Где в коде работать:
- `backend/app/api/v1/*`
- `backend/app/services/*`
- `backend/app/models/*`
Что должно получиться на выходе:
- endpoint-ы работают по контракту и видят только данные текущего пользователя.
Зависит от: от готовых моделей и auth-контекста `current_user`.
Можно делать параллельно: по доменам — projects/notes (Костя), tasks/files/tags/search (Андрей), frontend-интеграция уже готовых endpoint-ов (Маша).
Крайний срок: 13 мая 2026.

### Задача: провести техническую проверку backend
Зачем нужна: перед интеграцией в финал нужен быстрый регрессионный контроль.
Что обязательно прочитать:
- `docs/api/errors.md`
- `docs/README.md`
Где в коде работать:
- тесты в backend (или smoke-сценарии через curl)
Что должно получиться на выходе:
- есть минимальный проверочный набор, который подтверждает работоспособность основных сценариев.
Зависит от: от готовности основных endpoint-ов.
Можно делать параллельно: с frontend полировкой, с AI документацией, с Electron финальной сборкой.
Крайний срок: 14 мая 2026.

## Frontend

### Задача: подключить frontend к реальному API
Зачем нужна: сейчас часть интерфейса живет на mock-данных, это не годится для MVP.
Что обязательно прочитать:
- `docs/api/auth.md`
- `docs/api/projects.md`
- `docs/api/notes.md`
- `docs/api/tasks.md`
- `docs/api/files.md`
- `docs/api/search.md`
Где в коде работать:
- `frontend/lib/*`
- `frontend/app/*`
Что должно получиться на выходе:
- ключевые страницы берут данные из backend, а не из фейковых хранилищ.
Зависит от: по каждому экрану от соответствующего backend endpoint-а.
Можно делать параллельно: auth-часть можно начать сразу; проекты/заметки/задачи/файлы подключать по мере готовности backend блоков.
Крайний срок: 14 мая 2026.

### Задача: подготовить UI-состояния и ручную регрессию
Зачем нужна: даже с рабочим API пользовательский сценарий может ломаться на загрузках, ошибках и пустых состояниях.
Что обязательно прочитать:
- `docs/api/errors.md`
- `TASKS/Общий_бэклог.md`
Где в коде работать:
- страницы в `frontend/app/*`
- UI-компоненты в `frontend/components/*`
Что должно получиться на выходе:
- основные экраны имеют loading/error/empty состояния, и есть список найденных багов по демо-сценарию.
Зависит от: от появления реальных API вызовов на экранах.
Можно делать параллельно: с backend стабилизацией и с Electron UX-полировкой.
Крайний срок: 15 мая 2026.

## AI/LLM

### Задача: встроить AI-подсказки без риска для стабильности продукта
Зачем нужна: AI должен помогать пользователю, но продукт не должен падать без ключей.
Что обязательно прочитать:
- `backend/app/services/llm.py`
- `docs/architecture/overview.md`
Где в коде работать:
- `backend/app/services/llm.py`
- новый AI endpoint в `backend/app/api/v1/*`
Что должно получиться на выходе:
- AI endpoint работает и в режиме с ключами, и в fallback-режиме без ключей.
Зависит от: от auth middleware и базового backend каркаса endpoint-ов.
Можно делать параллельно: с backend доменными задачами и с frontend/API-клиентом до этапа UI-интеграции.
Крайний срок: 14 мая 2026.

Зачем нужна: по продуктовой задаче MVP должен запускаться как приложение, а не только в браузере.
Что обязательно прочитать:
- `TASKS/Общий_бэклог.md`
- текущие frontend/backend сценарии
Где в коде работать:
- папка desktop или frontend/electron-слой (по выбранной структуре)
Что должно получиться на выходе:
- Electron-приложение запускается, открывает рабочий продукт, проходит ключевой демо-сценарий.
Зависит от: dev-режим почти ни от чего не зависит; финальный demo зависит от готовности frontend/backend.
Можно делать параллельно: почти со всеми backend/frontend работами, кроме финальной демо-проверки.
Крайний срок: 15 мая 2026.

## Финальная приемка MVP

### Что считаем готовностью backend
- Основные endpoint-ы из `docs/api/*` реализованы.
- Ошибки отдаются в формате `{ detail, code }`.
- Пользователь не видит чужие данные.

### Что считаем готовностью frontend
- login/register работают через backend.
- Проекты, заметки, задачи, файлы и поиск работают без зависимости от mock CRUD.
- На ключевых экранах есть loading/error/empty состояния.

### Что считаем готовностью AI/LLM
- AI endpoint работает при наличии ключей.
- Без ключей работает fallback без падения приложения.

### Что считаем готовностью Electron
- Desktop-приложение запускается.
- Ключевые пользовательские сценарии проходят в Electron.

### Демонстрационный сценарий
1. Запустить backend и frontend, открыть Electron (или web-версию).
2. Зарегистрировать и залогинить пользователя.
3. Создать проект `Запуск MVP`.
4. Создать заметку, добавить тег `важно`.
5. Создать задачу `Подготовить демо` с высоким приоритетом.
6. Загрузить файл и скачать его обратно.
7. Открыть overlay и найти сущности через поиск.
8. Проверить AI-подсказку (LLM или fallback).
9. Обновить профиль, выйти и войти снова.
10. Перезапустить Electron и убедиться, что приложение стабильно.
