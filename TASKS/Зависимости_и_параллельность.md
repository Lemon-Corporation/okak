# Карта зависимостей и параллельной работы

## Фаза 1: стартовые задачи (можно делать сразу)

- Костя: выравнивание моделей `users/projects/notes`.
- Андрей: модели `tasks/files/tags` и связующие таблицы.
- Маша: API-клиент и мапперы DTO.
- Даша: карта экранов с mock-данными.
- Миша: контракт AI endpoint-а и fallback-дизайн.
- Александр: выбор архитектуры Electron и запуск dev-обертки.

Эти задачи можно делать параллельно, они почти не блокируют друг друга.

## Фаза 2: первая волна интеграции

- Backend auth (Костя) + единый формат ошибок (Андрей).
- Frontend auth-интеграция (Маша) + состояния auth-форм (Даша).
- Electron dev-режим и безопасные настройки (Саша).

Зависимости:
- frontend auth-код можно начать сразу, но финально закрыть только после готового backend auth.

## Фаза 3: доменные API и экраны

- Костя: projects + notes + проектные списки.
- Андрей: tags + tasks + files + search.
- Маша: по мере готовности endpoint-ов подключает экраны projects/notes/tasks/files/search.
- Даша: подхватывает UI-состояния для уже подключенных экранов.

Зависимости:
- `/projects` нужен раньше, чем полноценные notes/tasks flows.
- files flows зависят от готового `project_id` и авторизации.
- overlay search зависит от готового `/search`.

## Фаза 4: AI и desktop финализация

- Миша: AI endpoint + fallback + документация поведения.
- Маша: встраивает AI-кнопку в UI.
- Александр: доводит Electron UX, проверяет upload/download, готовит сборку.

Зависимости:
- AI UI зависит от готового AI endpoint.
- Electron финальная демонстрация зависит от стабильных frontend/backend сценариев.

## Фаза 5: финальная приемка

- Андрей: backend smoke-проверки.
- Даша: ручная регрессия по демо-сценарию.
- Костя и Андрей: сверка Swagger с docs.
- Александр: desktop demo run.

Зависимости:
- приемка начинается после закрытия критичных backend endpoint-ов и базовой frontend интеграции.

## Критический путь (что блокирует финиш MVP)

1. Модели и авторизация backend.
2. Projects/notes/tasks/files API.
3. Frontend интеграция этих API.
4. Проверка доступа и ошибок.
5. Финальный demo-flow в web и Electron.

## Что делать, если зависимость не готова

- Не ждать без дела: переключаться на параллельные задачи из своей роли.
- Явно фиксировать блокер в формате:
  - какая задача заблокирована;
  - чем заблокирована;
  - кто владелец блокирующей задачи;
  - когда ожидается разблокировка.
